# Agentry — Deployment Model

This document covers how Agentry is packaged and installed: the Helm chart contents, the cert-manager / trust-manager certificate inventory, the NetworkPolicy CNI prerequisite, and the two-tier on-ramp (gateway-only vs full agent lifecycle). For the system topology and component responsibilities, see [ARCHITECTURE.md](./ARCHITECTURE.md).

## Helm Chart Contents

Agentry ships as a Helm chart that installs:

- The CRDs (AgentClass, ModelProvider, Agent, AgentTask, AgentChannel), shipped in the chart's `crds/` directory — applied on `helm install`, never touched by `helm upgrade` (see [Upgrade and Migration](#upgrade-and-migration) for the explicit CRD upgrade step).
- The operator Deployment with RBAC, ServiceAccount, and leader election. Default `controller.replicas: 2` (overridable via the `controller.replicas` Helm value), PodDisruptionBudget (`minAvailable: 1`), `maxUnavailable: 1` rolling-update strategy, and pod anti-affinity (`topologyKey: kubernetes.io/hostname`) so the two replicas land on different nodes. Leader election picks one active replica; the second is a warm standby for failover and also serves the activator endpoint (the activator handler runs on **every** replica — see [ARCHITECTURE.md § Control Plane](./ARCHITECTURE.md#control-plane)), so two replicas keep the activator reachable across voluntary disruptions. Without a PDB, a multi-node drain or autoscaler downscale could evict both replicas simultaneously and surface as `controller_unavailable` 504s for any in-flight webhook to a hibernated Agent (see [GATEWAY_USER.md § Activator](./GATEWAY_USER.md#activator) step 5). The chart enforces a ≥2 floor with a template guard (Helm's `fail` function) that aborts rendering when `controller.replicas < 2`, mirroring `gateway.replicas`. The floor is operational, not correctness-driven — leader election and the activator both work with one replica — but a single-replica Controller breaks wake-on-demand availability during voluntary disruptions (drains) and single-replica involuntary failures, which contradicts the "hard control-plane dependency" framing in [ARCHITECTURE.md § The Agentry Gateway](./ARCHITECTURE.md#the-agentry-gateway).
- The Agentry Gateway Deployment with its RBAC, ServiceAccount, default `replicas: 2` (overridable via the `gateway.replicas` Helm value), PodDisruptionBudget (`minAvailable: 1`), and `maxUnavailable: 1` rolling update strategy. Two replicas is the floor: at one replica `minAvailable: 1` blocks all voluntary eviction (node drains stall), and `maxUnavailable: 1` rolling updates have no headroom — chart upgrades would briefly take the gateway offline for both LLM proxy and inbound webhook traffic. The multi-replica state model in [ARCHITECTURE.md § The Agentry Gateway](./ARCHITECTURE.md#the-agentry-gateway) (spend ConfigMap exchange, divide-by-replicas rate buckets, controller activity fan-out) degrades gracefully to one replica, so the floor is operational, not correctness-driven. The chart enforces this with a `fail` template guard that aborts rendering when `gateway.replicas < 2`, so an operator cannot accidentally drop below the floor. The gateway's operational settings are exposed as Helm values — notably `gateway.maxFallbackDepth` (default: `3`), which sets the `AGENTRY_MAX_FALLBACK_DEPTH` environment variable on the gateway Deployment and controls the maximum fallback chain depth for LLM provider routing (see [Fallback Logic](./GATEWAY_LLM.md#fallback-logic)). Several additional optional values govern network reachability and runtime tuning:
  - `gateway.callbackUrl.allowlist` — list of DNS-name suffixes or CIDR blocks. When set, **replaces** the default deny-internal rule for `AgentChannel.spec.webhook.callbackUrl` with an explicit allowlist; the [AgentChannelReconciler](./CONTROLLER_RECONCILERS.md#agentchannelreconciler) and the gateway's delivery-time re-check admit only hosts that match one of the configured entries. Leaving it unset preserves the default: `https://` only, with loopback, link-local, RFC1918, unique-local IPv6, and cloud-metadata IPs denied. See [API_RESOURCES.md § Cross-Resource Validation rule 22](./API_RESOURCES.md#cross-resource-validation).
  - `controller.networkPolicy.dnsSelector` — object of the shape `{ namespaceLabels: {...}, podLabels: {...} }` used as the `namespaceSelector` and `podSelector` for the DNS egress rule on every synthesized per-agent NetworkPolicy. Defaults to `{ namespaceLabels: { "kubernetes.io/metadata.name": "kube-system" }, podLabels: { "k8s-app": "kube-dns" } }`, which matches kubeadm, EKS, GKE, AKS, and the upstream CoreDNS chart. Override for clusters that run DNS in a non-standard namespace or with custom labels — see [SECURITY.md § Protecting agent containers from LLM provider access](./SECURITY.md#protecting-agent-containers-from-llm-provider-access).
  - `gateway.externalHostnames` — list of additional DNS names appended to the `agentry-gateway-tls` Certificate's SAN list. Required when the User Gateway is exposed via TLS pass-through Ingress so that external clients see a cert whose SAN matches the public hostname they dialed. Unset by default — backend re-encrypt Ingress works without it because the Ingress controller dials the in-cluster Service DNS, which is already in the default SAN set. See [GATEWAY_USER.md § TLS and Ingress](./GATEWAY_USER.md#tls-and-ingress).
  - `gateway.channelHealthWindow` — duration string (default `5m`) controlling the rolling window over which the gateway evaluates `AgentChannel.status.conditions[type=PlatformConnected]`. Within this window, a channel reports `True` if any inbound request succeeded, `False` if the window holds only failures, and `Unknown` (`reason=NoRecentTraffic`) if no observations exist on a replica that has been up the full window. Tune larger for low-traffic webhooks (avoid spurious `Unknown` flapping), smaller for high-traffic channels where staler "last success" data is undesirable. See [GATEWAY_USER.md § Channel Health Tracking](./GATEWAY_USER.md#channel-health-tracking).
  - `gateway.agentDeliveryConnectTimeout` — duration string (default `1s`) bounding the gateway's TCP connect attempt when delivering `POST /v1/message` to an Agent Service. On iptables-mode kube-proxy, hibernated Agents fail-fast via RST and this timeout is essentially unused; on IPVS / Cilium kube-proxy replacement / eBPF data paths that drop packets to empty-endpoint ClusterIPs, this is the per-wake latency cap before the gateway falls through to the activator wake. See [ARCHITECTURE.md § Per-Agent and Per-Task Child Resources](./ARCHITECTURE.md#per-agent-and-per-task-child-resources) and [GATEWAY_USER.md § Activator](./GATEWAY_USER.md#activator).
  - `gateway.maxResponseBodyBytes` — byte count (default `900Ki`) capping the size of an agent's webhook response, applied uniformly in sync and async modes. Set well below the Kubernetes ~1 MiB ConfigMap object cap so async responses fit in the per-request `agentry-async-{requestId}` ConfigMap with envelope headroom, and to bound the gateway's per-request memory footprint while it buffers the response in either mode. The payload is stored as text in the ConfigMap's `data` field (JSON envelope), never in `binaryData` — base64 encoding would inflate 900Ki past the object cap and silently break the headroom math. The gateway returns `response_too_large` to the caller (sync) or to `callbackUrl` / the polling endpoint (async) for over-cap responses. See [ARCHITECTURE.md § The Agentry Gateway](./ARCHITECTURE.md#the-agentry-gateway) and [GATEWAY_USER.md § Request Flow](./GATEWAY_USER.md#user-gateway--request-flow).
  - `gateway.maxMessageBodyBytes` — byte count (default `1Mi`) capping inbound webhook bodies on `:8080`; over-cap POSTs get `413` at the listener level, before path resolution and auth. See [GATEWAY_USER.md § Request Flow step 2](./GATEWAY_USER.md#user-gateway--request-flow).
  - `gateway.maxLLMRequestBodyBytes` — byte count (default `4Mi`) capping inbound LLM-proxy request bodies on `:8443`; over-cap requests get `413 request_too_large` before namespace identification. See [API_ENDPOINTS.md § LLM Proxy Endpoints](./API_ENDPOINTS.md#llm-proxy-endpoints).
  - `gateway.providerFirstByteTimeout` — duration string (default `120s`) bounding each upstream LLM-provider attempt from connection start through the first response byte. This is the per-attempt timeout behind the fallback table's "timeout before any response bytes" trigger and the `504 provider_timeout` exhaustion mapping; the same value also applies mid-stream as an idle-bytes bound between SSE chunks, so an upstream that stalls without closing cannot hold gateway and agent connections open indefinitely. The default is deliberately generous — LLM providers can legitimately take a minute-plus to first byte on long prompts — and it bounds the worst-case pre-error wait at `maxFallbackDepth × providerFirstByteTimeout`. See [GATEWAY_LLM.md § Fallback Logic](./GATEWAY_LLM.md#fallback-logic).
  - `gateway.syncDeliveryDeadline` — duration string (default `30s`) bounding total sync-mode wall-clock (wake + delivery retries + agent processing); exceeded → `504 sync_deadline_exceeded`. See [GATEWAY_USER.md § Request Flow step 6a](./GATEWAY_USER.md#user-gateway--request-flow).
  - `gateway.agentReadTimeout` / `gateway.callbackReadTimeout` — duration strings (both default `10s`) bounding the per-attempt read of an agent's `/v1/message` response and of a callback receiver's response, respectively. Together with the retry schedules below they define the ~31s–~71s retry-budget wall-clock cited in [API_ENDPOINTS.md § Async Webhook Response](./API_ENDPOINTS.md#async-webhook-response-gateway-managed).
  - `gateway.agentDeliveryRetryBackoff` / `gateway.callbackRetryBackoff` — backoff schedules (both default `1s,5s,25s`; 4 attempts total) for the agent-delivery and callback-delivery pipelines. Independently tunable; they merely share a default. The async response-`Patch` pipeline reuses `gateway.callbackRetryBackoff`.
  - `gateway.healthPort` — port (default `8081`) for the gateway's internal kubelet-probe listener (`/healthz`, `/readyz`; TLS, no client auth). See [GATEWAY_LLM.md § Gateway Readiness](./GATEWAY_LLM.md#gateway-readiness).
- Prometheus metrics are served on dedicated ports — controller `:8080/metrics`, gateway `:9090/metrics`, both unauthenticated in-cluster. The chart does **not** ship `ServiceMonitor` / `PodMonitor` manifests; scrape integration (and a NetworkPolicy admitting only the Prometheus ServiceAccount, if desired) is left to the platform team. See [OBSERVABILITY.md § Metrics](./OBSERVABILITY.md#metrics).
- The two ClusterIP `Service`s whose stable DNS names the whole SAN/endpoint design hangs on: `agentry-gateway` (`:8080` user listener, `:8443` LLM/internal mTLS listener, `:9090` metrics) and `agentry-controller` (`:9443` activator, `:8080` metrics). Every certificate SAN, `$AGENTRY_GATEWAY_ENDPOINT` value, and internal RPC in the other docs assumes these names in `agentry-system`. The chart targets the `agentry-system` namespace; install with `--namespace agentry-system --create-namespace`. The optional `agentry-upstream-ca` ConfigMap ([GATEWAY_LLM.md § Upstream TLS Configuration](./GATEWAY_LLM.md#upstream-tls-configuration)) is operator-supplied, not chart-templated — the chart only documents its name.
- A single default AgentClass (`standard`) that platform teams can customize or delete. It leaves `runtimeClassName` unset, so Pods run under the cluster's default container runtime (runc in practice) — stock clusters define no `RuntimeClass` objects at all, so pinning a named one (even `runc`) would fail Pod admission with "RuntimeClass not found" everywhere it isn't explicitly created. A `sandboxed` example manifest (gVisor [`RuntimeClass`](./SECURITY.md#runtimeclass)) ships in the chart's `examples/` directory; operators apply it after confirming the matching `RuntimeClass` is installed on the cluster, since shipping it as a live default would put any Agent that selected it into [`Degraded`](./CONTROLLER_LIFECYCLE.md#agent-persistent-mode) on clusters without gVisor.
- Optional: a sample ModelProvider manifest stub (keys not included) as a starting template.

## Certificate Lifecycle

**cert-manager and trust-manager are required dependencies.** The chart does not install the cert-manager or trust-manager controllers themselves (teams with an existing cert-manager deployment reuse them); it ships the `ClusterIssuer`, `Certificate`, and `Bundle` resources Agentry needs. See [SECURITY.md § In-cluster TLS](./SECURITY.md#in-cluster-tls-bidirectional) for the full TLS topology.

- A self-signed `ClusterIssuer` (`agentry-selfsigned`) creates a `Certificate` for the Agentry CA. The CA `Certificate` and the `agentry-ca` Secret it writes live in cert-manager's **cluster resource namespace** — Helm value `certManager.clusterResourceNamespace`, default `"cert-manager"` — not in `agentry-system`: a CA `ClusterIssuer` resolves `spec.ca.secretName` only in that namespace (cert-manager's `--cluster-resource-namespace` flag), and trust-manager likewise reads `Bundle` sources only from its trust namespace (`--trust-namespace`, default `cert-manager`). Operators who run either controller with a non-default namespace must set the value to match, or issuance fails cluster-wide with `SecretNotFound`.
- A `ClusterIssuer` (`agentry-ca-issuer`) sourcing from the `agentry-ca` Secret in the cluster resource namespace signs all Agentry-issued leaf certs (a `ClusterIssuer` is used so per-namespace `Certificate` resources can reference the same signing key).
- A `Certificate` for the gateway serving cert (`agentry-gateway-tls`) used by both listeners — the LLM listener on port 8443 and the User listener on port 8080 — both serving TLS from the same cert. **Despite the conventional HTTP association of port 8080, the User listener is TLS-only**; an Ingress fronting it must use HTTPS as its backend protocol. External webhook traffic arrives via Ingress configured for backend re-encrypt (or TLS pass-through); there is no plaintext listener on the gateway. See [GATEWAY_USER.md § TLS and Ingress](./GATEWAY_USER.md#tls-and-ingress).
- A `Certificate` for the controller's activator endpoint (`agentry-controller-tls`).
- One `Certificate` per Agent, created by the [AgentReconciler](./CONTROLLER_RECONCILERS.md#agentreconciler) at provisioning time and owned by the Agent resource via ownerRef — see [SECURITY.md § Lifecycle of an agent TLS serving certificate](./SECURITY.md#lifecycle-of-an-agent-tls-serving-certificate).
- One `Certificate` per AgentTask, created by the [AgentTaskReconciler](./CONTROLLER_RECONCILERS.md#agenttaskreconciler) at provisioning time and owned by the AgentTask resource via ownerRef — see [SECURITY.md § Lifecycle of an AgentTask TLS client certificate](./SECURITY.md#lifecycle-of-an-agenttask-tls-client-certificate).
- A `trust-manager` `Bundle` resource (`agentry-ca`) that projects the Agentry CA as a ConfigMap into every non-system user namespace (including future ones added after install). Agent and AgentTask Pods mount this ConfigMap at `/var/run/agentry/ca.crt` to verify the gateway's TLS cert. Platform teams that need a tighter projection can override the selector via the Helm value `trustManager.bundleSelector` (an object with `matchLabels` / `matchExpressions` keys passed verbatim into the `Bundle`'s `target.namespaceSelector`).

Admission webhooks are not used; the cert-manager dependency is solely for TLS lifecycle management.

## Tiered On-Ramp

The Helm chart supports a tiered on-ramp:

1. **Gateway only**: install the chart, configure a [ModelProvider](./API_RESOURCES.md#modelprovider), and point existing workloads at the gateway for LLM traffic and spend tracking. No AgentClass, Agent, AgentTask, or AgentChannel resources need to be created in this tier. Existing workloads authenticate to the gateway using their own projected ServiceAccount tokens — no client certificate is required in this tier (see [SECURITY.md § Agent→Gateway Authentication](./SECURITY.md#agentgateway-authentication) Mode 2 and [Namespace Identification](./GATEWAY_LLM.md#namespace-identification)). The token **must** be minted for audience `agentry-gateway` via a `serviceAccountToken` projected volume (`audience: agentry-gateway`) — the gateway's `TokenReview` names that audience, so a Pod's default `kubernetes.default.svc`-audience token is rejected. A workload that skips this step gets `401` on every call. Because Agentry does not mutate non-managed Pods in this tier, the workload manifest must hard-code or template the gateway URL itself — `https://agentry-gateway.agentry-system.svc:8443` is the in-cluster Service DNS. The controller injects `$AGENTRY_GATEWAY_ENDPOINT` only into full-lifecycle-tier Pods. Existing workloads must also mount the `agentry-ca` ConfigMap (projected by trust-manager into every non-system namespace — see the trust-manager `Bundle` description above) and configure their HTTP client to trust it; otherwise calls to the gateway fail TLS verification. Access control in this tier is governed by `ModelProvider.spec.allowedNamespaces` plus `spec.models` only — AgentClass `allowedProviders` does not apply because there is no Agent resource to reconcile against. Platform teams who need class-scoped provider policy must use the full lifecycle tier.
2. **Full agent lifecycle**: configure [AgentClasses](./API_RESOURCES.md#agentclass), deploy Agents and AgentTasks with hibernation and wake-on-demand, and connect them to user-facing channels via AgentChannels (webhook in v1). Channel integration is included in this tier because wake-on-demand requires a channel to be fully testable. Agentry-managed Pods authenticate via mTLS using per-agent certificates issued by cert-manager. The LLM Gateway enforces the full routing chain (Agent → AgentClass `allowedProviders` → ModelProvider `allowedNamespaces`/`models`) for this tier — see [Provider Routing](./GATEWAY_LLM.md#provider-routing).

## Upgrade and Migration

This section is scoped to what `v1alpha1` promises. Multi-version CRD conversion machinery — conversion webhooks, storage-version migration, deprecation windows — is deliberately deferred until the API graduates past alpha: while `v1alpha1` is the only served and stored version there is nothing for such machinery to convert, and shipping it early would contradict the no-admission-webhook posture in [CONTROLLER_RECONCILERS.md § Operator Structure](./CONTROLLER_RECONCILERS.md#operator-structure) for zero benefit.

**Rolling upgrade order.** CRDs live in the chart's `crds/` directory, which `helm install` applies but `helm upgrade` **never touches** — CRD schema changes must be applied explicitly with `kubectl apply --server-side -f crds/` *before* running `helm upgrade`, and that command is the documented first step of every chart upgrade. (Keeping CRDs in `crds/` rather than `templates/` also means `helm uninstall` never deletes them — deleting a CRD cascade-deletes every CR of that kind cluster-wide.) Helm's kind-sorted apply is not a rollout barrier either: the controller and gateway Deployments are applied together and roll **concurrently**, so there is no ordering guarantee between the controller observing new CRD fields and the gateway consuming what the controller writes (status fields, budget `_canonical`, per-task RBAC) — the version-skew tolerance described below, not apply order, is what makes the rollout safe. Both Deployments roll with `maxUnavailable: 1` under their PDBs, so one replica of each stays serving throughout; wake-on-demand and the LLM proxy remain available across the upgrade. **Version-skew tolerance is one chart version, and only for the duration of an in-progress rollout**: the internal contracts (activity and channel-health response bodies, the budget ConfigMap shape, the activator wire contract, the per-request async ConfigMap labels) evolve additively within a minor version so a mixed old/new controller↔gateway pair works mid-rollout. Running mixed versions as a steady state is unsupported.

**CRD schema evolution (within v1alpha1).** Changes are additive-only: new optional fields with reconcile-time defaulting (per [API_RESOURCES.md § Defaulting](./API_RESOURCES.md#defaulting)), new condition reasons, new enum values. Anything breaking ships as a replacement alpha version in a new chart release, with the migration documented in that release's notes — the alpha contract is `kubectl get -o yaml` → adapt → re-apply, not automated conversion. `v1` API stability is explicitly not a goal for the initial release (see [API_RESOURCES.md](./API_RESOURCES.md)).

**Helm chart upgrades.** Values whose change has workload-visible effects are the ones to review before an upgrade: `gateway.replicas` (rate-limit buckets re-divide on the next refill cycle — see [GATEWAY_LLM.md § Rate Limiting](./GATEWAY_LLM.md#rate-limiting)), `controller.networkPolicy.dnsSelector` (regenerates every synthesized per-agent NetworkPolicy), `gateway.channelHealthWindow` (changes `PlatformConnected` flapping behavior), and the body-size caps (in-flight requests sized between old and new caps change fate). The chart's `fail` template guards (`replicas ≥ 2` floors) abort rendering on known-bad values. A gateway rollout also resets in-memory activity state: expect idle/hibernation transitions to defer for `idleTimeout` afterwards, per [CONTROLLER_LIFECYCLE.md § Activity Detection](./CONTROLLER_LIFECYCLE.md#activity-detection) — schedule chart upgrades accordingly on clusters with multi-hour idle timeouts.

**ModelProvider credential rotation, end-to-end.** (1) Platform engineer updates the credential Secret in `agentry-system`. (2) The gateway's Secret watch refreshes in-memory credentials without a restart ([SECURITY.md § Lifecycle of an LLM API key](./SECURITY.md#lifecycle-of-an-llm-api-key)); in-flight requests complete on the old key. (3) The next ModelProviderReconciler health probe validates the new key — a bad rotation surfaces as `Ready=False, reason=CredentialsInvalid` within one probe interval (default 60s), plus a `Warning` event from any live traffic that hits upstream `401`/`403` and falls back ([GATEWAY_LLM.md § Fallback triggers](./GATEWAY_LLM.md#fallback-triggers)). (4) Audit: the Secret update lands in the Kubernetes audit log (enable `RequestResponse` level for `agentry-system` Secrets per [SECURITY.md § Recommendations](./SECURITY.md#recommendations-for-deployment)); confirm cut-over on the provider's own key-usage dashboard. No Pod restarts at any step, in either tier.

**Breaking Agent / AgentTask / AgentChannel spec changes (within alpha)** are replace-not-migrate: let in-flight AgentTasks run to completion (or delete them), hibernate or delete Agents, apply the new CRDs, re-apply updated manifests. Agent state survives the replace through the PVC path — `pvcRetention: Retain` on delete, or snapshot-and-remount via [`persistence.existingClaim`](./API_RESOURCES.md#agent). Channel receivers see a bounded gap: webhook paths return `401` while the AgentChannel is absent (indistinguishable from an unregistered path, per the [401 contract](./API_ENDPOINTS.md#post-channelsnamespacechannel-path-external--webhook-caller)), and stored async responses survive independently in `agentry-system` until their 1-hour TTL unless the channel is deleted (the finalizer sweeps them — see [CONTROLLER_LIFECYCLE.md § Finalizers](./CONTROLLER_LIFECYCLE.md#finalizers)).
